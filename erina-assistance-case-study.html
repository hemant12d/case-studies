Technical Deep Dive

Engineering That Powers Every Roadside Moment.

📍

Google Maps Integration

Draggable pin location selection on service booking and warehouse/hub creation. Autocomplete search with Places API. Drop location for towing with real-time distance calculation.

Places API · Geocoding · Distance Matrix
🎫

Dynamic Subscription Schema

Subscription categories define field schemas (number, dropdown, yes/no, percentage, time). Each subscription package inherits and customizes these fields. Fully dynamic at runtime — no hardcoded fields.

MongoDB Dynamic Schema · Admin-Configurable
🔗

Public Service URL Generation

After every service booking (by customer or admin), a unique shareable public URL is generated. Admin can share via WhatsApp. Third-party service recipients can view status without login.

JWT-Signed Tokens · Public Read API
💬

WhatsApp Service Channel

Customers can initiate service booking via WhatsApp. Admin can push service status, public URL, and updates via WhatsApp API. Seamless fallback for non-app users.

WhatsApp Business API · Webhook
📦

5-Level Product Hierarchy

Category → Brand → Model → Fuel Type → Product. Each level is independently CRUD-managed in admin. Product discovery on the customer side drills down through all 5 levels.

NestJS Tree Entity · MongoDB Refs
🏭

Warehouse → Hub Architecture

Warehouses contain multiple hubs. Service requests and product deliveries are dispatched from the nearest hub. Hub panel users manage their own queue independently.

Geospatial Queries · Hub Assignment
🔔

Multi-Audience Notifications

Admin creates push notifications with audience targeting (all, specific segment) and scheduling. Bull queue delivers via Firebase FCM with retry logic. Notification page with unread filter on customer side.

Firebase FCM · Bull · Scheduled Push
📱

Mobile OTP Auth

Mobile-number OTP authentication for customer app and hub panel. Device session management, token rotation, and hub-level role system (manager, assistant, technician, field technician).

SMS OTP · JWT · Role Guards

Engineering Decisions

Real Problems. Engineered Solutions.

// CHALLENGE 01

Dynamic Subscription Fields

⚠ ProblemCar, Bike, UPS, and Solar subscriptions all have completely different benefit structures. Hardcoding fields for each type would make the system unmaintainable.
✓ SolutionAdmin defines a field schema per category (type: number/dropdown/yes-no/percentage/time). Each subscription inherits the schema and stores values against dynamic keys in MongoDB. Zero hardcoded fields.
// CHALLENGE 02

Towing Distance Pricing

⚠ ProblemTowing pricing depends on real distance between pickup and drop location. Showing the correct price before payment required accurate KM calculation without a paid routing API call per user.
✓ SolutionGoogle Maps Distance Matrix API called on drop location confirmation. Result cached per (origin, destination) pair. Vehicle-type multipliers applied server-side. Price displayed with GST before payment.
// CHALLENGE 03

Admin Booking on Behalf of User

⚠ ProblemWhen a customer calls support, the admin must create a service booking on their behalf and instantly share status access — without the customer needing a login.
✓ SolutionAdmin booking flow mirrors the customer flow but with a customer selector. On save, a signed JWT-based public URL is generated. Admin shares via WhatsApp. Anyone with the URL can view service status without auth.
// CHALLENGE 04

Multi-Panel Role Architecture

⚠ ProblemThree different panels (customer, admin, hub) with different roles (owner, admin, hub manager, technician, delivery) all hitting the same NestJS backend required careful guard design.
✓ SolutionJWT payload includes panel type + role. NestJS guards check panel context first, then role permissions. Module-level guards protect entire controller groups. Hub staff access scoped to their assigned hub's data only.
// CHALLENGE 05

SEO + Content CMS

⚠ ProblemEvery public-facing page needs independently manageable meta titles, descriptions, and OG tags. Hardcoding these meant a deployment for every copy change.
✓ SolutionAdmin can update SEO data per page slug via a simple form. Next.js fetches the meta record at build-time for static pages and at request-time for dynamic pages. Content changes deploy in seconds, no code touch.
// CHALLENGE 06

Warehouse → Hub → Assignment Chain

⚠ ProblemService requests need to be dispatched from the closest hub. Hub staff should only see their own hub's queue, not all requests.
✓ SolutionService request record stores assigned warehouse and hub IDs. Hub panel queries filter by hubId from the authenticated user's profile. Admin assigns the hub on service creation; hub staff accept/reject from their own queue.
