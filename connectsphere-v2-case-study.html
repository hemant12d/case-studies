Project Overview

One Platform. Every Way Teams Communicate.

ConnectSphere is a full-stack community and communication platform built for teams, creator communities, and social groups — combining the best of Discord's structured workspace with Instagram's ephemeral content and WhatsApp's directness.

Built entirely from scratch, with no off-the-shelf communication SDKs. Every feature — from WebSocket message routing to WebRTC signaling to Redis-backed Blip expiry — is purpose-engineered for performance, scale, and reliability.

The Problem

Four Apps to Do One Team's Job

Teams were duct-taping Discord + Slack + WhatsApp + Instagram together. ConnectSphere was built to end that.

01

Tool Fragmentation

Discord for channels, Slack for work, WhatsApp for quick messages, Instagram for stories — no single platform handled all communication modes for a modern team.

02

No Unified RBAC

Existing platforms offered either no permissions or coarse-grained roles. There was no solution with group-level + channel-level + user-level permission inheritance in one tool.

03

Mobile Gap

Discord and Slack are desktop-first. The mobile experience feels bolted on. Teams needed a mobile-first platform that worked as well on a phone as in a browser.

04

No Ephemeral Content

No communication platform for teams offered story-style ephemeral content. Instagram Stories exist in social context only — not inside workspaces or community channels.

05

WebSocket Instability

Off-the-shelf solutions couldn't handle multi-instance WebSocket scaling. Message drops, ghost notifications, and desync plagued existing open-source alternatives.

06

No Integrated Calling

Third-party video call SDKs are expensive, slow, and disconnected from the chat context. A native WebRTC solution was needed tightly integrated with groups and channels.

Engineering Solution

Architected for Discord-Class Scale

A complete MERN ecosystem, designed modular from day one, with every realtime component custom-engineered.

Full-Stack Ecosystem, Zero Dependencies on Communication SDKs

Every realtime feature in ConnectSphere is custom-built. WebSocket infrastructure uses Socket.IO with a Redis adapter for multi-instance horizontal scaling — no sticky sessions, no single-server bottleneck. WebRTC signaling is handled by a custom NestJS gateway, no Twilio or Agora dependency.

The NestJS backend is split into 9 domain modules (Auth, Groups, Channels, Permissions, Messaging, Blips, Calls, Members, Profiles), each with its own service layer, DTOs, guards, and event emitters. The architecture is microservice-ready with zero structural refactoring needed.

NestJS Modules Socket.IO + Redis WebRTC Signaling Bull Queues AWS S3 + CloudFront Firebase FCM
⚡

Event-Driven Core

NestJS EventEmitter decouples domain events — message sent, Blip created, call started — from side effects like notifications and audit logs.

🛡️

3-Layer RBAC

Permission inheritance: Group role → Channel override → Per-user exception. Evaluated at API guard level, not in business logic.

⏱️

Blip TTL Engine

Redis key expiry fires an event caught by a Bull worker that cleans up storage, removes DB references, and purges CDN cache.

☁️

Media Pipeline

Pre-signed S3 uploads bypass the backend entirely. Lambda processes and compresses media, CloudFront serves it globally at edge.

Technical Deep Dive

Engineering That Goes Beyond the Surface

🌐

WebSocket Scaling

Socket.IO with Redis adapter broadcasts across all Node.js instances. No sticky sessions. Each server subscribes to Redis channels and routes only to its own socket connections.

Socket.IO · Redis Adapter
🛡️

RBAC Permission Engine

Three-layer permission evaluation: Group role defaults → Channel-level overrides → Per-user exceptions. Evaluated via NestJS Guards before any business logic executes.

Guards · Decorators · CASL
✨

Blip TTL Engine

Blips are created with Redis key TTL set to 24 hours. On key expiry, a keyspace notification fires a Bull worker that deletes DB records, purges S3 media, and invalidates CloudFront cache.

Redis TTL · Bull · Lambda
📹

WebRTC Signaling

Custom NestJS WebSocket gateway handles STUN/TURN negotiation, ICE candidate exchange, and offer/answer flow. No Twilio or Agora. Fully owned, zero per-minute cost.

WebRTC · ICE · TURN Server
🗄️

Polyglot Persistence

MongoDB for messages, events, and profiles (flexible schema). PostgreSQL for auth, RBAC, and billing (relational integrity). Redis for caching, pub/sub, sessions, and Blip TTLs.

MongoDB · PostgreSQL · Redis
🔐

JWT Token Architecture

15-minute access tokens with rotating refresh tokens in HttpOnly cookies. Token family tracking prevents reuse attacks. Device-level session revocation on suspicious activity.

JWT · Refresh Rotation · Guards
📡

Presence System

User presence (Online/Away/DND/Offline) stored in Redis sorted sets with heartbeat TTLs. Bulk presence reads use MGET. Status changes published via Socket.IO rooms to only interested subscribers.

Redis Sets · Heartbeat · WS Rooms
☁️

Media CDN Pipeline

Pre-signed S3 URLs let clients upload directly, bypassing the API. Lambda processes images (WebP conversion, thumbnails) and videos (HLS transcode). CloudFront serves globally with <80ms P99.

S3 · Lambda · CloudFront

📁 Media Upload Path

  • S3 event triggers Lambda processor
  • Lambda compresses, generates thumbnails
  • CloudFront cache invalidated via API
  • CDN serves media at edge globally

🔔 Notification Path

  • Domain event emitted on message/mention
  • Notification service picks up via EventEmitter
  • In-app: delivered via Socket.IO room
  • Push: queued in Bull for Firebase FCM
  • Bull worker retries on FCM failure
  • Dead-letter queue captures final failures

Engineering Decisions

Hard Problems, Thoughtful Solutions

// CHALLENGE 01

Multi-Instance WebSocket Scaling

⚠ ProblemSocket.IO has no shared state across Node.js instances. Users on different pods couldn't receive each other's messages.
✓ SolutionRedis adapter makes every Socket.IO instance subscribe to a shared Redis pub/sub channel. Messages are published once and fan-out to all instances holding relevant sockets.
// CHALLENGE 02

3-Layer Permission Inheritance

⚠ ProblemFlat role systems couldn't express "this user has this permission in this group, but not in this specific channel."
✓ SolutionPermission evaluation chain: Group role → Channel override → User exception. All three layers resolved in a single DB query per request via NestJS Guard, cached in Redis per (user, channel) pair.
// CHALLENGE 03

Blip Lifecycle Management

⚠ ProblemEphemeral content needs to expire precisely at 24h, clean up media from S3, remove DB references, and not leave orphaned CDN cache entries.
✓ SolutionRedis key with 24h TTL. Keyspace notification triggers Bull job on expiry. Job deletes MongoDB record, fires S3 delete, invokes CloudFront invalidation. Verified end-to-end in staging.
// CHALLENGE 04

WebRTC Without Third-Party SDK

⚠ ProblemTwilio Video and Agora charge per-minute fees that make the unit economics non-viable at scale. Building signaling from scratch seemed risky.
✓ SolutionCustom NestJS WebSocket gateway handles the full ICE/SDP exchange. COTURN self-hosted on AWS for TURN relay. Zero per-minute cost. Full control over call routing and quality metrics.
// CHALLENGE 05

Message Threading Data Model

⚠ ProblemNested reply threads in MongoDB can grow unboundedly and become expensive to query without a careful schema design.
✓ SolutionFlat message documents with a threadRootId field. Thread messages are their own top-level documents pointing to a root. Single compound index on (channelId, threadRootId, createdAt) enables efficient thread pagination.
// CHALLENGE 06

Presence at Scale

⚠ ProblemPolling 1000+ members' online status on every channel view is too expensive. A naive pub/sub approach would flood the event bus.
✓ SolutionRedis sorted set stores all online users with heartbeat timestamps as scores. Member list subscribes to a presence room — only members in the same groups receive status change events. Heartbeat TTL auto-expires stale sessions.
