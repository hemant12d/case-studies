The Problem

A customer is waiting at the machine. The system can't make them wait too.

A print job only proceeds after the document owner approves it on WhatsApp. The moment they tap “Allow,” the kiosk needs to show the payment screen, then print — while a real person stands at the counter. The naive approaches all failed this bar in their own way.

Rejected approaches

Why polling and push notifications didn't work

  • Polling the backend every few seconds added visible lag and wasted requests at scale
  • Push notifications (FCM) had unpredictable delivery timing — unacceptable for a waiting customer
  • One-directional channels still needed a second path for the machine to report results
  • None guaranteed that a paid job survived a dropped connection
What the system needed

The real requirements behind the feature

  • Sub-second command delivery while the customer is present
  • Two-way: commands down to the kiosk, results back up to the backend
  • A guarantee that a paid-but-undelivered job is never silently lost
  • Maintainable by a single engineer, on infrastructure already in place
The Decision

Six mechanisms evaluated. One chosen for a reason, not a reflex.

Rather than reaching for the familiar, every viable transport was weighed against the actual constraints: a waiting customer, a two-way conversation, and a solo maintainer.

Socket.IO won on the two things that mattered most: real-time, bidirectional delivery for a waiting customer, and a manageable operational footprint on infrastructure already running. MQTT is the better answer at large fleet scale — and the design leaves room to migrate to it without rewriting the flow.

The Flow

One continuous loop, from OTP entry to printed page.

Each step hands cleanly to the next — and every command is persisted before it's sent.

01

Kiosk connects & identifies

On boot, the machine opens a live connection and joins its own private channel. Any commands queued while it was offline replay instantly.

channel · kiosk:{machineId}
02

Customer enters their code

The kiosk submits the OTP; the backend verifies it and sends a WhatsApp approval request to the document's owner.

03

Owner approves on WhatsApp

The moment they tap “Allow,” the backend pushes payment options to the kiosk in real time — QR, UPI, or checkout.

event · payment_options
04

Payment confirmed → print dispatched

On payment, the backend persists the print command, then pushes it to the machine with every file and setting it needs.

event · print_command
05

Machine prints & reports back

The kiosk prints, then reports the outcome. The result is recorded, the owner is notified on WhatsApp, and the loop closes.

The hard part

A real connection drops. A paid job must never drop with it.

Shop Wi-Fi is unreliable. The difference between a toy and a production system is what happens when the socket dies at the exact moment a command is sent. The dispatch layer was built so that a dropped connection is a recoverable blip — never a lost, paid job.

Persist first

Every command is written to a durable queue before it's emitted — so it survives a drop.

Replay on reconnect

When a machine comes back online, any undelivered commands are automatically re-sent.

Money lives in the database

Payment truth is held in persistent storage, not the live cache — a cache failure can never lose a paid job.

Under the Hood

The decisions that separate a demo from a deployable system.

  • ›
    Per-machine channelsEach kiosk joins an isolated room, so a command can target exactly one machine with no cross-talk.
  • ›
    Persist-before-emitCommands are queued durably before sending, making delivery resilient to connection loss.
  • ›
    Reconnect replayThe reconnect handshake re-delivers anything missed while the machine was dark.
  • ›
    Result-closes-commandA command stays recoverable until the machine confirms the actual print result — not just receipt.
  • ›
    Append-only print historyEvery outcome is its own immutable record, enabling re-prints and per-machine reporting.
