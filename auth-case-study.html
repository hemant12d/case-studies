01 / Auth Flows

Four Core Flows,
Zero Gaps

Every authentication path is covered — from first OTP to token rotation to admin-level revocation.

FLOW_01 · REGISTER

Registration

Phone-first onboarding with Firebase Identity Toolkit. User created on first successful OTP verification.

POST /auth/get-otp → rate limit check
Firebase sends SMS via Identity Toolkit
sessionInfo stored in Redis (10min TTL)
POST /auth/verify-otp → user created
Session + token pair generated
FLOW_02 · LOGIN

Multi-Device Login

Same phone can authenticate from up to 5 devices simultaneously. Each device gets its own session and token family.

deviceId (UUID) sent from client
Existing session for device revoked first
New session created with device context
Access token (15m) + Refresh token (30d)
Session cached in Redis immediately
FLOW_03 · REFRESH

Token Rotation

Every refresh invalidates the old token and issues a new pair atomically. Replay attacks trigger full family revocation.

Old token hash matched in Firestore
Firestore transaction — atomic rotation
usedAt set → new token created
Reuse detected → entire family revoked
New access + refresh pair returned
FLOW_04 · LOGOUT

Single + All Devices

Logout from current device or all devices simultaneously. Admin can also revoke any specific session remotely.

Session deleted from Redis immediately
All tokens for session revoked in Firestore
logout-all → batch revoke all sessions
Admin revoke → specific session + tokens
Audit log written for every action

03 / Security Layer

Defense in Depth

Security built into every layer — not bolted on after the fact.

🔐

JWT + Session Hybrid

Stateless JWT for performance, session store for immediate revocation. Best of both worlds.

🔁

Token Family Rotation

Tokens share a family UUID. Reuse of a consumed token triggers full family revocation — no stolen token can survive.

🛡️

Redis Rate Limiting

Sliding window per phone + per IP. OTPs capped at 5/hour per number. Survives restarts and multi-instance deploys.

🔒

Token Hashing

Raw refresh tokens never stored. SHA-256 hashed before persistence — database leak can't expose live tokens.

⏱️

Progressive Lockout

5 failed OTP attempts triggers a 15-minute account lock. Tracked in Firestore, checked before every OTP send.

👮

RBAC + Admin Revoke

Role-based guards on every route. Admins can revoke any specific user session in real-time without restart.

04 / Edge Cases

Every Edge Case,
Handled

Production systems fail in unexpected ways. These are anticipated and resolved before they can surface.

5 Simultaneous Devices

Each device has its own session + token family. Logging in from a new device revokes the old session for that device only.

Token Stolen but Valid

Admin revokes the session. Redis key deleted → next request misses cache → Firestore shows isActive: false → 401 immediately.

OTP Resend Abuse

Sliding window counter per phone + IP in Redis. 5 OTPs/hour per number, 15 per IP. Prevents SMS bill attacks.

Concurrent Refresh Requests

Firestore transaction locks the old token. Second concurrent request finds usedAt already set → TokenReuseException → family revoked.

User Banned with Active Session

Ban check happens in requestOtp. Active sessions blocked on next JWT validation when session service checks user status.

Firebase OTP Succeeds, DB Fails

OTP session cleared from Redis. User must request a new OTP. No orphaned tokens or partial state left behind.

Clock Drift on Token Expiry

JWT library handles signature validation with standard exp claim. Session store uses server timestamps exclusively.

Guest + Logged-in Same Endpoint

OptionalAuthGuard checks Bearer token first, then x-guest-session-id header. Both paths validated. Neither present → 401.
