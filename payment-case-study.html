01 / State Machine

Every Payment Has a State.
Every State is Tracked.

Payments don't binary succeed or fail. A state machine tracks every transition — preventing double captures, stale retries, and ghost orders.

FLOW_01 · ORDER CREATION

Idempotent Order

Orders are deduplicated at DB level before hitting Razorpay. Same request twice — same order returned, no duplicate charge.

Check DB for existing order (userId + context)
Existing + CREATED state → return it
POST /orders to Razorpay API
Save order to Firestore (CREATED state)
Return razorpayOrderId to client
FLOW_02 · PAYMENT CAPTURE

Verification Flow

Client-side payment completion triggers server-side HMAC signature verification before any order fulfillment happens.

Client sends razorpayPaymentId + signature
HMAC-SHA256 verification (secret + ids)
Signature mismatch → 400, order untouched
Fetch payment from Razorpay API to confirm
Update order state → CAPTURED in Firestore
FLOW_03 · WEBHOOK HANDLING

Idempotent Webhooks

Razorpay can deliver the same webhook multiple times. Every event is deduplicated by eventId before processing business logic.

Verify X-Razorpay-Signature header first
Check eventId in DB — already processed?
Yes → 200 OK, no duplicate processing
No → process event, write to DB atomically
Handle: payment.captured, payment.failed
FLOW_04 · FAILURE RECOVERY

Failed Payment Recovery

Network timeouts and user drop-offs don't leave orphaned orders. Failed orders are recoverable and tracked clearly.

Order in CREATED/PROCESSING state
Webhook delivers payment.failed event
Order state → FAILED in Firestore
Client can retry → new order created
Old FAILED order retained for audit

05 / Edge Cases

Production Breaks
in Unexpected Ways.

Every scenario that causes silent failures in a naive integration — handled explicitly.

Duplicate Order Creation

DB checked for existing CREATED order before calling Razorpay. Same user + same context returns existing order — no second charge.

Webhook Delivered Multiple Times

Razorpay retries failed webhooks. EventId stored in DB on first processing — subsequent deliveries return 200 without reprocessing.

Webhook Before Client Confirmation

Webhook can arrive before the client confirms. State machine handles both paths — first writer wins, second is a no-op.

Payment Captured, DB Write Fails

Reconciliation catches this. Razorpay shows captured, internal DB shows processing — auto-heal corrects the state on next sync.

Network Timeout After Order Creation

Order exists in Razorpay but client never received ID. Client retries → idempotency check finds existing order → returned safely.

Forged Signature Attempt

HMAC-SHA256 verification runs before any DB operation. Invalid signature → 400 immediately. No order state touched.

User Closes Browser Mid-Payment

Order stays in PROCESSING state. Webhook delivers payment.failed → state updated to FAILED. User can retry with fresh order.

Order Expired, Payment Attempted

Razorpay order expires after 15 minutes by default. Expired orders are detected on verify — client told to create a fresh order.
