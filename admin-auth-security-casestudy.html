02

Security Holes — The Critical Section

Admin auth is where a single hole becomes catastrophic. This is the heart of the analysis.

⬆️
Privilege Escalation
Role tampering via request body · namespace crossing
CRITICAL
Scenario — Role Tampering
If any endpoint trusts a client-supplied role field, a Super Admin could craft a request claiming tech_admin privileges. "Immutable in schema" is not the same as "enforced at the write path."
Scenario — Namespace Crossing
A Super Admin's JWT used against /api/tech/* routes. Middleware must check role against namespace — not just "valid JWT."
Mitigations
  • Role is never read from request body; derived only from authenticated JWT
  • On registration, role is hardcoded server-side — never passed in
  • Write-time guard rejects any payload containing a role field entirely
  • Namespace check (token role === namespace role) runs before every handler, hard reject otherwise
🌱
Seeded Tech Admin — Bootstrap Risk
Seed credentials · password fallback · last-admin lockout
HIGH
Scenario — Credentials in Source Control
If the seed script contains Tech Admin phone/details and is committed, anyone with repo access knows the highest-privilege account.
Scenario — Password Hash Fallback
A bcrypt passwordHash "fallback for internal tooling" on the highest-privilege account is a second auth path — and second auth paths are second attack surfaces.
Scenario — Last Admin Lockout
If the single seeded Tech Admin loses phone access (lost SIM, number ported), there is NO in-app recovery — by design.
Mitigations
  • Seed values from environment / secrets manager — never hardcoded in the script
  • Seed script is idempotent (re-running doesn't duplicate or reset state)
  • Seed at least two Tech Admin accounts on controlled numbers
  • Document DB-intervention recovery procedure — not tribal knowledge
🚩
Flag to Senior: Gate the password fallback behind an env flag, OFF by default in production. Every auth path you don't ship is an attack surface you don't have to defend. Decision needed: is the password fallback required for launch, or deferred?
🔢
OTP — The Primary Auth Factor
Brute force · replay · enumeration · SIM swap
CRITICAL
Scenario — OTP Brute Force
6-digit OTP = 1,000,000 combinations. 3 attempts → 15-min lockout is good — but can an attacker request a fresh OTP repeatedly to reset the attempt counter and keep guessing?
  • Lockout must be per-phone within a rolling window — not just per-OTP-request
  • Cap OTP requests at ≤5 per phone per hour, independent of attempt count

Scenario — Replay & Enumeration
If a verified OTP isn't immediately consumed, it can be replayed. Differing error messages for "phone not found" vs "wrong OTP" allow admin phone enumeration.
  • OTP is single-use: deleted/consumed the instant it verifies
  • Identical generic response for phone-not-found, wrong-OTP, and lockout-active
  • Never reveal whether a phone is a registered admin account

Scenario — SIM Swap
OTP over SMS is vulnerable to SIM-swap attacks. For Tech Admin accounts, this is the single biggest real-world attack vector.
🚩
Flag to Senior: For launch, SMS OTP is acceptable — documented risk. For the future, evaluate TOTP (authenticator app) or hardware key as a second factor specifically for Tech Admin. At minimum, alert on Tech Admin login from a new device.
🪙
Session & Token Security
Token theft · live session abuse · restriction bypass
HIGH
Scenario — Stolen Token (24h window)
24h expiry, no refresh token. A stolen JWT is valid for up to 24 hours with no way to detect misuse mid-window.
Scenario — Restriction Without Session Kill
A Super Admin is restricted mid-flight. Their existing JWT may still be valid. What about removal — does an existing valid token at moment of deletion continue to work?
Mitigations
  • Token stored hashed in session record
  • Middleware checks active session record on every request — not just JWT signature
  • isDeleted and isRestricted checked on every request — a removed account is rejected on its very next API call
  • Per-request DB fetch cannot be skipped for performance without reintroducing this hole
💳
The Money Surface — Super Admin Payouts
Payout detail tampering · revenue share out of bounds
CRITICAL
Scenario — Payout Redirection
A compromised Super Admin (or escalated attacker) changes an Admin's payoutDetails to redirect money to an attacker-controlled bank account / UPI.
Scenario — Revenue Share Out of Bounds
No stated validation on revenueSharePercent. A value of 150%, -10%, or 0% could break downstream payout math.
Mitigations
  • Every change to payoutDetails / revenueSharePercent is audit-logged with before + after values
  • Validate: 0 ≤ revenueSharePercent ≤ 100
  • Log who set it, when it was changed, and from what previous value
🚩
Decision needed: Should payout changes require a second approval (Tech Admin sign-off)? For launch: audit-log it. For scale: consider dual-control.
📋
Audit Log Integrity
Silent failures · not truly append-only · accountability illusion
MEDIUM
Scenario — Action Succeeds, Audit Fails
Restrict a Super Admin succeeds, but the history insert fails (DB blip). Now there's a restriction with no record of who did it or why. If the audit log can be tampered with or fails silently, accountability is an illusion.
Mitigations
  • Audit write and the action are as close to atomic as the DB allows
  • If audit write fails, that is a logged ERROR — the one place where real alerting is non-negotiable
  • History collections have no update/delete methods — append-only enforced in code, not just convention
  • For launch: action proceeds, audit failure logged loudly. For scale: transactional write.
03

Failure Modes — Production Breakage

Things that won't show in testing but will break in production.

FAILURE · 01
Duplicate Registration (Double-Submit)
Tech Admin taps "Register Super Admin" twice or network retries. Two super_admins with the same phone could be created.
Unique constraint on phone before insert; phone is the natural unique key
FAILURE · 02
Abandoned Profile After OTP Verify
OTP verified → fill profile → create. User abandons after OTP but before submit. Dangling verification state.
Account created only at final submit. OTP verification alone creates nothing. No half-records.
FAILURE · 03
Restrict During Mid-Flight Action
Super Admin registers a Machine when Tech Admin restricts them. Does the in-flight request complete?
Middleware checks isRestricted at request start. In-flight may complete; next one is blocked. Document explicitly.
FAILURE · 04
Removed Super Admin — Orphaned Ownership
Soft-delete a Super Admin. Their registeredBy references on Admins/Machines now point to a deleted account.
registeredBy is a historical snapshot, not a live FK. Admins/Machines created by them stay active — deliberate, not a bug.
FAILURE · 05
Machine Phase 1 Done, Phase 2 Never Happens
Super Admin registers a machine (Phase 1). Kiosk app never syncs (Phase 2). Machine sits half-registered indefinitely.
Status reflects this: active but phase2CompletedAt = null. Report "machines pending first sync." Not a launch blocker.
FAILURE · 06
linkedMachines Array Drift
Machine insert succeeds but the Admin.linkedMachines array push fails. Machine exists but isn't linked.
Order operations so array push is last. On failure, machine is unlinked but recoverable. Log the inconsistency at launch.
FAILURE · 07
Clock Skew on OTP / Session Expiry
Expiry computed on different servers with drifting clocks. OTPs expire early or late; sessions misjudged.
All expiry computed server-side from a single time source. Never trust client timestamps.
FAILURE · 08
History Collection Grows Unbounded
Every login, view, and action is logged forever. Over time, query performance degrades without indexing.
Index on actorId, targetId, date. Archival / TTL strategy deferred but flagged for post-launch.
FAILURE · 09
Phone Number Reuse After Removal
A removed Super Admin's phone is reused to register a new Super Admin. The soft-deleted record still holds that phone.
Unique-phone check filters isDeleted=false. Old record preserved for audit; phone can be re-registered.
FAILURE · 10
Restriction Reason Missing Despite "Mandatory"
Spec says reason is mandatory for restrict/remove. If validation is only client-side, a crafted API request omits it.
Server-side validation rejects restrict/remove without a non-empty reason. Mandatory = enforced at the API, not just the UI.

Launch-Critical vs Deferred

✅ LAUNCH-CRITICAL Cannot ship without
🔐Phone + OTP login, both roles
📦Per-role session collections + JWT
🛡️Restriction middleware (role-match, isDeleted, isRestricted on every request)
🔒Role derived from token, never request body
👤Register Super Admin / Admin / Machine flows
⛔Restrict / Unrestrict / Remove flows
📝Mandatory reason enforced server-side
📋Audit log writes for all actions
⏱️OTP single-use + per-phone rate limiting
🤐Uniform login error responses (no enumeration)
🌱Seed script from env (not hardcoded)
➕Append-only history (no update/delete methods)
🔄Before/after value capture on every field change
⏭️ DEFERRED Build against real need
🔑MFA / TOTP on Tech Admin
🔔New-device login alerting
🚪"Revoke all sessions" on restrict/remove
✌️Dual-control approval for payout changes
🗄️Audit log archival / TTL strategy
🔗Transactional audit writes
💰Payout-change second approval
⛔ Explicit Non-Goals
Admin role internals (Module 03) · Operator management · Cross-namespace calls

Open Decisions for Senior Sign-Off

□
Password fallback on Tech Admin — ship or defer?
Defer. Gate behind env flag, OFF by default. Every auth path you don't ship is an attack surface you don't have to defend.
□
OTP rate limit model — confirm per-phone rolling window + max OTP requests per hour
5 OTP requests/hour per phone. Lockout per-phone, not per-request. Reset counter is the attack vector.
□
Reuse user OTP/JWT utilities, but separate session store — confirm this boundary is enforced
Reuse OTP-generation + JWT-signing utilities. Do NOT reuse user session collection.
□
Phone reuse after soft-delete — allow re-registration of removed account's phone?
Yes — allow reuse. Filter isDeleted=false in unique-phone pre-check. Old record preserved for audit.
□
registeredBy as historical snapshot — confirm no cascade on admin/machine removal
Deliberate choice. registeredBy is audit data, not a live foreign key. Admins/Machines survive the removal of their creator.
□
revenueSharePercent bounds + payout-change audit with old values — confirm validation range
Validate 0 ≤ revenueSharePercent ≤ 100 server-side. Audit captures before + after values on every change.
□
Seed: minimum 2 Tech Admins on controlled numbers + documented DB-recovery procedure
At least 2 seeded accounts. Recovery procedure written in runbook — not tribal knowledge.
□
All time-based fields are server-authoritative — confirm no client timestamps are trusted anywhere
All expiry and timestamp fields computed server-side from a single time source.
